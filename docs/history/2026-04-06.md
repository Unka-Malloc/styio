## Checkpoint（F.2.1：Stream Zip 不支持来源的误分类收敛）

### 当前状态

- 发现可触发路径：
  - `a = [1, 2, 3]` 后参与 `a >> #(n) & ["x", "y", "z"] >> #(s) => { ... }`；
  - 原行为命中 `StyioNotImplemented("unsupported stream zip lowering")`，顶层归类 `RuntimeError`（退出码 `5`）。
- 先失败后修复（TDD 书签）：
  - 新增 `StyioDiagnostics.StreamZipUnsupportedSourceReportsTypeError`；
  - 初始红灯（exit `5`）后转绿（exit `4`）。
- 修复内容：
  - `SGStreamZip` 默认兜底分支从 `StyioNotImplemented` 改为 `StyioTypeError`；
  - 文案明确当前支持边界：`list literal` 与 `@file stream`。
- 回归结果：
  - `ctest -L styio_pipeline` 全绿（25/25）；
  - `ctest -L security` 全绿（78/78）；
  - `ctest -L milestone` 全绿（92/92）；
  - `ctest -L soak_smoke` 全绿（6/6）。
- ADR 落地：`docs/adr/ADR-0063-stream-zip-unsupported-source-typeerror.md`。

### 下一步

1. F.2 继续：扫描剩余 `StyioNotImplemented` 可触发路径（优先 codegen），逐条做“红灯用例 + 分类收敛”。
2. F.1 协同：保持 text/jsonl 在错误分类与退出码上的一致性，防止同类错误漂移回 `RuntimeError`。
3. D.5 联动：将该失败样本纳入回归模板，作为长周期恢复时的快速健康检查。

### 复现命令

## Checkpoint（F.2.2：活跃流水线 `NotImplemented` 清零收口）

### 当前状态

- 代码收口：

  - `ToStyioIR.cpp` 中 16 处 `StyioNotImplemented` 全部替换为 `StyioTypeError`；
  - 叠加 F.2.1 的 `CodeGenG.cpp` 收口后，活跃链路（Parser/Analyzer/CodeGen）不再抛 `StyioNotImplemented`。
- 行为回归补齐：
  - `StyioParserEngine.HashIteratorMatchForwardChainReturnsParseError` 增强断言：输出含 `Styio.ParseError`，不含 `Styio.NotImplemented`；
